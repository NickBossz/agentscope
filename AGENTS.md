# AGENTS.md — AgentScope

This file defines the instructions that AI coding agents must follow when working on AgentScope.

## 1. Project Context

AgentScope is an observability platform for AI agents. It collects traces and spans from AI applications and displays prompts, responses, tool calls, token usage, estimated cost, errors, evaluations, and execution trees.

Read `docs/PRD.md` before implementing features. The PRD is the source of truth for product scope and behavior.

## 2. Main Stack

Use the following stack unless an existing implementation clearly requires otherwise:

- Next.js with App Router;
- TypeScript with strict mode;
- Bun as runtime and package manager;
- Elysia for the backend API;
- PostgreSQL;
- Drizzle ORM;
- Redis;
- OpenTelemetry;
- Tailwind CSS;
- shadcn/ui;
- TanStack Query;
- TanStack Table;
- React Flow;
- Recharts;
- Zod or Elysia schemas for validation;
- Docker and Docker Compose.

Do not replace these technologies without a documented technical reason.

```text

  api/
  worker/

  database/

  telemetry/
  shared/
```

Responsibilities:

### `apps/web`

### `apps/api`

- REST API.

- Query endpoints.

### `apps/worker`

- Event processing.
- Cost calculation.
- Aggregations.
- Evaluation jobs.
- Alert processing.

### `packages/database`

- Drizzle schemas.
- Migrations.
- Database client.

### `packages/shared`

- Shared types.
- Constants.
- Schemas.
- Utility functions.

### `packages/sdk-node`

### `packages/ui`

- Reusable UI components.

## 4. General Development Rules

- Prefer small, focused changes.
- Preserve existing architecture and conventions.
- Do not rewrite unrelated code.
- Avoid premature abstractions.
- Do not add dependencies when the existing stack already solves the problem.
- Never commit secrets, API keys, tokens, or credentials.
- Never expose raw API keys after their creation response.
- Never log full prompts, responses, authorization headers, or secrets without explicit safe configuration.
- Keep all tenant-sensitive queries scoped by organization and project.
- Use UTC for timestamps.
- Store monetary values in a precise decimal or integer representation, never floating-point values.
- Store durations in milliseconds unless the existing schema defines another unit.
- Use stable IDs generated by the server or validated client-generated IDs.

## 5. TypeScript Standards

- Keep `strict: true`.
- Avoid `any`.
- Prefer `unknown` and validate before use.
- Add explicit return types to exported functions.
- Use discriminated unions for event and span types.
- Use domain-specific types for IDs where practical.
- Keep shared API contracts in `packages/shared`.
- Do not duplicate schemas between frontend, API, and worker.
- Validate all external data at runtime.

Example:

```ts
export type SpanType =
  | "agent"
  | "llm"
  | "tool"
  | "retrieval"
  | "embedding"
  | "reranking"
  | "database"
  | "http"
  | "code"
  | "evaluation"
  | "custom";
```

## 6. API Rules

- Follow REST conventions.
- Version public APIs under `/v1`.
- Use consistent response envelopes.
- Return machine-readable error codes.
- Validate headers, path parameters, query parameters, and bodies.
- Use appropriate HTTP status codes.
- Make ingestion endpoints idempotent.
- Keep ingestion fast and move expensive work to workers.
- Do not calculate analytics synchronously during ingestion.
- Add rate limiting to public ingestion endpoints.
- Authenticate ingestion using project API keys.
- Authenticate dashboard APIs using the application session.

Recommended error format:

```json
{
  "error": {
    "code": "INVALID_TRACE_PAYLOAD",
    "message": "The trace payload is invalid.",
    "details": {}
  }
}
```

## 7. Ingestion Rules

The ingestion flow should be:

```text
request
  -> authenticate API key
  -> validate payload
  -> redact configured fields
  -> verify idempotency
  -> enqueue event
  -> return quickly
```

Requirements:

- Support individual and batch events.
- Use an event ID for deduplication.
- Preserve trace and span relationships.
- Accept out-of-order spans when possible.
- Do not assume a parent span is already stored.
- Record ingestion failures with safe metadata.
- The monitored application must not depend on AgentScope availability.

## 8. Worker Rules

Workers are responsible for:

- persisting queued events;
- calculating duration;
- calculating token totals;
- estimating cost;
- updating trace summaries;
- generating analytics aggregates;
- running evaluations;
- triggering alerts.

Workers must:

- be idempotent;
- support retries;
- use bounded retry counts;
- move permanently failing events to a dead-letter mechanism;
- log structured error information;
- never process the same event twice in a way that duplicates data.

## 9. Database Rules

- Use Drizzle migrations for schema changes.
- Do not modify production schemas manually.
- Add indexes for frequent filters and joins.
- Avoid N+1 queries.
- Use transactions for operations that must be atomic.
- Store JSON only for flexible metadata, not for fields frequently filtered or aggregated.
- Keep tenant ownership explicit on sensitive tables.
- Add unique constraints for idempotency keys and externally supplied event IDs.
- Prefer soft deletion for projects and API keys where auditability matters.

Every query involving tenant data must be scoped by at least one of:

- organization ID;
- project ID.

## 10. Security Rules

- Hash API keys using a secure cryptographic hash.
- Show raw API keys only once.
- Use constant-time comparison where appropriate.
- Redact authorization headers.
- Sanitize user-controlled content before rendering.
- Enforce authorization server-side.
- Do not rely on hidden UI elements for access control.
- Apply secure cookie settings in production.
- Protect mutation endpoints against CSRF when session authentication is used.
- Validate redirect URLs.
- Avoid exposing stack traces to clients.
- Add audit logs for API key creation, rotation, and revocation.

## 11. Privacy and Redaction

AgentScope may receive sensitive prompts and outputs.

Implementation requirements:

- Support disabling prompt capture.
- Support disabling response capture.
- Redact known secret patterns.
- Provide configurable field redaction.
- Never persist API authorization headers.
- Keep raw and redacted values clearly separated if raw storage is ever supported.
- Respect project retention settings.
- Make deletion operations complete and auditable.

## 12. Frontend Rules

- Use Server Components by default.
- Use Client Components only when interaction requires them.
- Use TanStack Query for client-side server state.
- Use URL query parameters for shareable filters.
- Keep tables accessible and keyboard-friendly.
- Handle loading, empty, error, and partial-data states.
- Do not render large prompt or response payloads without truncation or virtualization.
- Use monospace formatting for prompts, JSON, errors, and tool payloads.
- Keep critical trace information visible without opening multiple dialogs.
- Avoid unnecessary animations in dense observability screens.

## 13. Execution Tree Rules

The execution tree is a core product feature.

- Build the tree from `spanId` and `parentSpanId`.
- Handle missing parents gracefully.
- Handle out-of-order spans.
- Detect cycles and prevent infinite rendering.
- Clearly distinguish span types.
- Clearly highlight failed and running spans.
- Display duration, tokens, and cost where available.
- Selecting a node must open the corresponding span details.
- Large traces must remain usable through collapsing, lazy rendering, or virtualization.
- Do not depend on node array order to infer hierarchy.

## 14. Cost Calculation Rules

Cost calculation must be deterministic and testable.

Use:

```text
input cost = input tokens × input price
output cost = output tokens × output price
cached cost = cached tokens × cached input price
total cost = input cost + output cost + cached cost + external costs
```

Requirements:

- Associate prices with provider, model, and effective date.
- Preserve the price version used for a calculation.
- Do not silently recalculate historical costs when prices change.
- Use decimal-safe arithmetic.
- Treat missing pricing as `unknown`, not zero.

## 15. OpenTelemetry Rules

- Instrument the API, workers, Redis, and database.
- Propagate trace context across HTTP and queue boundaries.
- Use semantic conventions where possible.
- Avoid recording sensitive prompt and response content as span attributes by default.
- Keep internal platform traces separate from customer agent traces.
- Configure sampling through environment variables.

## 16. Redis Rules

Redis may be used for:

- queues or streams;
- rate limiting;
- caching;
- idempotency windows;
- temporary trace updates;
- real-time event distribution.

Do not use Redis as the permanent source of truth.

All cached values must have an expiration policy unless permanence is explicitly required.

## 17. Testing Requirements

Every meaningful feature should include tests at the appropriate level.

### Unit tests

Use for:

- cost calculation;
- token aggregation;
- tree construction;
- redaction;
- validation;
- status derivation;
- percentile helpers.

### Integration tests

Use for:

- API authentication;
- ingestion;
- idempotency;
- database repositories;
- worker processing;
- multi-tenant isolation.

### End-to-end tests

Critical flows:

1. Create an account.
2. Create a project.
3. Generate an API key.
4. Send a trace with nested spans.
5. Open the trace in the dashboard.
6. Inspect prompts, tools, costs, and errors.

Do not consider a feature complete when its critical behavior is untested.

## 18. Code Quality Checks

Before considering a task complete, run the relevant commands:

```bash
bun install
bun run lint
bun run typecheck
bun run test
bun run build
```

When database changes are included, also run migration checks.

When Docker changes are included, verify:

```bash
docker compose config
docker compose up --build
```

If a command cannot be run, clearly state why.

## 19. Environment Variables

Use `.env.example` to document required variables.

Expected categories:

```text
DATABASE_URL=
REDIS_URL=
AUTH_SECRET=
APP_URL=
API_URL=
OTEL_EXPORTER_OTLP_ENDPOINT=
OTEL_SERVICE_NAME=
TRACE_ENCRYPTION_KEY=
```

Rules:

- Never commit `.env` files containing secrets.
- Validate required environment variables at startup.
- Fail with a clear message when a required variable is missing.

## 20. Docker Rules

The local environment should be startable with Docker Compose.

Expected services:

- web;
- api;
- worker;
- postgres;
- redis;
- otel-collector.

ClickHouse may be introduced after the PostgreSQL MVP.

Docker images should:

- use pinned major versions;
- use multi-stage builds where useful;
- run as a non-root user when practical;
- include health checks;
- avoid embedding secrets.

## 21. Documentation Rules

Update documentation when changing:

- architecture;
- public APIs;
- environment variables;
- database schema;
- local setup;
- ingestion payloads;
- SDK usage.

Public endpoints and SDK methods must include examples.

## 22. Task Workflow for AI Agents

Before coding:

1. Read `docs/PRD.md`.
2. Inspect the current repository structure.
3. Identify existing conventions.
4. Locate related schemas, routes, components, and tests.
5. Define the smallest complete implementation.

During coding:

1. Keep changes scoped.
2. Reuse existing shared types and components.
3. Add validation and authorization.
4. Add or update tests.
5. Handle error and empty states.
6. Avoid unrelated refactors.

After coding:

1. Run formatting, linting, type checking, tests, and build.
2. Review the diff for secrets and accidental changes.
3. Confirm multi-tenant scoping.
4. Confirm API compatibility.
5. Summarize files changed and remaining limitations.

## 23. Definition of Done

A task is complete only when:

- the requested behavior works;
- types are correct;
- external input is validated;
- authorization is enforced;
- tenant isolation is preserved;
- errors are handled;
- tests cover critical logic;
- documentation is updated when necessary;
- lint, typecheck, tests, and build pass;
- no secrets or sensitive data were introduced.

## 24. Priority Order

When requirements conflict, follow this priority:

1. Security and tenant isolation.
2. Data correctness and idempotency.
3. PRD requirements.
4. Existing repository conventions.
5. Performance.
6. Developer convenience.

## 25. Important Product Constraints

- Start with PostgreSQL for the MVP.
- Add ClickHouse only when analytical volume justifies it.
- Keep ingestion asynchronous.
- AgentScope outages must not break monitored applications.
- Preserve historical pricing used in cost calculations.
- The execution tree must handle incomplete and out-of-order data.
- Prompts and responses are sensitive and must be optional to store.
